# Политика безопасности и доступа (SQLite, `bookstore_lab.db`)

## 1. Ограничение модели SQLite

SQLite — **встраиваемая** СУБД: в одном файле БД **нет** серверных учётных записей и команд `GRANT` / `REVOKE` на уровне движка, как в PostgreSQL. Разграничение «кто что может» в чистом SQL достигается так:

- **Физическая защита файла** БД (права ОС, только нужные пользователи/службы).
- **Логическая модель** таблиц `roles` и `db_users` (роли и пользователи приложения).
- **Представления** `v_readonly_*` и `v_audit_readonly` для сценария «только чтение» без прямого доступа к базовым таблицам (если клиентское ПО настроено открывать только эти объекты).
- **Триггеры** на изменение данных и таблица `audit_log` для журнала.

Формулировка ТЗ «серверная СУБД» для SQLite обычно трактуется как **размещение файла БД на сервере** и доступ по сети через приложение; сам движок остаётся встраиваемым.

## 2. Роли и пользователи (логическая модель)

| Роль       | Смысл |
|-----------|--------|
| `admin`   | Полный DML по предметным таблицам и чтение `audit_log`. |
| `librarian` | Авторы и книги — DML; клиенты и заказы — по политике только чтение (соблюдается в приложении / отдельными скриптами). |
| `sales`   | Клиенты и заказы — DML; каталог — чтение. |
| `readonly` | Работа только с представлениями `v_readonly_catalog`, `v_readonly_orders`, `v_audit_readonly`. |

Учётные записи в `db_users` связаны с `roles`; пароли в учебном проекте — заглушки, в реальной системе — только хэши и проверка в приложении.

## 3. Матрица прав (целевое поведение клиента)

| Объект / операция | readonly | sales | librarian | admin |
|-------------------|----------|-------|-------------|-------|
| `authors`, `books` | SELECT (через `v_readonly_catalog`) | SELECT | INSERT/UPDATE/DELETE | полный DML |
| `customers`, `orders`, `order_items` | SELECT (`v_readonly_orders`) | полный DML | SELECT | полный DML |
| `audit_log` | SELECT (`v_audit_readonly`) | нет | нет | SELECT |

Жёсткое принуждение к роли средствами одного только SQLite без приложения **невозможно**; политика фиксирует ожидаемые права и объекты для клиента.

## 4. Целостность

- `PRAGMA foreign_keys = ON`, внешние ключи, `CHECK`, `UNIQUE`.
- Триггер `trg_books_stock_check` запрещает отрицательный остаток.

## 5. Аудит

- Таблица `audit_log` и триггеры после INSERT/UPDATE/DELETE по `books`, `orders`, `order_items`.
- Дополнительно при отладке: `PRAGMA trace` / профилирование на соединении (не хранится в БД).

## 6. Резервное копирование

- Скрипт `scripts/backup_sqlite.ps1` — команда `.backup` для согласованной копии при WAL.
- Допустима копия файла при остановленной записи в БД.

## 7. Пересмотр

При смене схемы обновлять скрипты в `sqlite/` и этот документ.
